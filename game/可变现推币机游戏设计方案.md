# 可变现推币机游戏设计方案

## 执行结论

推荐先做 **Web 单机 MVP**，用 `Vite + TypeScript + Phaser` 快速验证推币手感、数值循环和广告变现模型。确认可玩性后，再迁移到微信小游戏接入广告、支付和云端存档。

关键合规边界：不做现金返还、实物兑换、有价奖品兑换、概率型付费抽奖或任何可被理解为赌博/博彩的机制。变现来自广告、去广告、虚拟币包、通行证、皮肤和关卡包。

## 1. 定位

做一款“街机推币机 + 收集养成 + 广告/内购变现”的休闲游戏。核心体验是玩家投币、推板推动硬币、硬币掉落后获得游戏内奖励，再用奖励解锁机器、皮肤、任务和收藏品。

重要边界：不设计现金返还、实物兑换、有价奖品兑换、概率型付费抽奖或任何可被理解为赌博/博彩的机制。变现来自广告、去广告、虚拟币包、通行证、皮肤和关卡包。

## 2. 推荐落地路线

### 首选：Web 单机 MVP

适合先验证玩法手感和广告变现模型。

- 引擎：Phaser 3 + Matter.js
- 运行方式：H5 页面，本地/服务器均可部署
- 数据：localStorage 起步，后续接后端
- 变现：Web 广告位、激励视频广告 SDK、买断去广告
- 优点：开发快、调试快、可直接上 Web
- 缺点：微信生态变现能力弱于小游戏

注意：推币机强依赖旋转、摩擦、堆叠和高密度刚体挤压，不使用 Phaser Arcade Physics。Arcade Physics 更适合轻量碰撞，不适合推币机核心手感。

### 第二阶段：微信小游戏

适合验证后做分发和微信广告变现。

- 引擎：Cocos Creator 或 LayaAir，或者 Phaser 适配小游戏运行环境
- 广告：Banner、插屏、激励视频
- 支付：虚拟道具内购需走微信小游戏合规支付能力
- 数据：微信登录 + 云开发/自建后端
- 优点：广告变现路径成熟、社交传播方便
- 缺点：审核、适配、包体和合规成本更高

## 3. 合规变现策略

### 可以做

- 激励视频：看广告获得免费投币次数、复活、双倍离线收益、任务刷新。
- 插屏广告：关卡结算后、机器升级完成后、每日首次领取后，频率受控。
- Banner 广告：主界面底部、商店页底部。
- 内购：购买游戏内金币、体力、外观、月卡、通行证。
- 去广告：一次性买断，保留激励视频可选入口。
- 皮肤：硬币样式、推板样式、机器主题、背景音乐。
- 通行证：每日任务和赛季任务给固定奖励，不做随机付费奖池。

### 不建议做

- 现金红包提现。
- 实物礼品兑换。
- 付费抽奖、盲盒、转盘、概率宝箱。
- 玩家之间交易代币。
- 用“中奖”“返现”“提现”等表达包装掉落奖励。

### 微信审核安全锁

- 文案去敏感化：界面不使用“金币爆率”“以小博大”“赢取”“中奖”“返现”“提现”等字眼。
- 收集品包装为“考古碎片”“机器图鉴”“主题收藏”，避开“抽奖”“奖池”“开奖”等概念。
- 任务系统和成长系统必须前置，让游戏第一感知是“物理机制 + 养成目标”，不是单纯投币模拟器。
- 新手引导第一屏展示主线任务，例如“修复第一台街机”“收集 3 枚主题零件”“升级推板稳定器”。
- 结算文案使用“获得”“收集”“解锁”“升级”，不使用“赢”“赌”“爆”“翻倍暴击”等刺激博彩联想的词。

## 4. 核心玩法循环

1. 玩家进入推币机。
2. 系统每隔一段时间恢复免费投币次数。
3. 玩家点击投币按钮，硬币从顶部落入台面。
4. 推板周期性前后运动，推动硬币群。
5. 硬币从前方掉落，按落点判定收益。
6. 玩家获得游戏内金币、经验、收藏碎片或任务进度。
7. 玩家用金币升级机器参数或解锁新主题。
8. 免费次数耗尽后，玩家可以等待恢复、看激励视频或购买虚拟币包。

## 5. 关键机制

### 推币台

- 台面宽度固定，左右边缘为无效掉落区。
- 前方中间为奖励槽，前方两侧为普通掉落。
- 推板以固定速度往复运动。
- 硬币有碰撞、堆叠和摩擦。
- 每局限制最大硬币数量，超过上限时清理最早生成的硬币，避免性能失控。

### Matter.js 手感参数

- 硬币使用圆形刚体，启用旋转，禁止使用 Arcade Physics。
- `restitution` 调低到 `0-0.1`，避免硬币像皮球一样弹跳。
- `friction` 稍高，建议从 `0.08-0.2` 调起，让硬币群有粘滞感。
- `frictionAir` 稍高，建议从 `0.01-0.04` 调起，降低硬币落地后乱滑。
- 硬币密度不要过低，否则推板推挤没有厚重感。
- 推板使用较厚的静态/运动刚体，厚度要大于视觉厚度，降低高速投币时穿透概率。
- 投币冷却要限制生成频率；高频投币时使用对象池，避免 Matter 世界刚体数量失控。
- 每局硬币上限建议 `120-180`，移动端低端机可降到 `80-120`。

### 防穿透策略

- 推板刚体厚度做大，碰撞体向运动方向额外扩展。
- 限制单帧最大速度，避免硬币高速穿过推板或墙体。
- Matter step 使用固定时间步长，避免低帧率时物理大步进。
- 对投币生成点做安全检测，避免硬币直接生成在推板或墙体内部。
- 掉出世界的硬币及时销毁或回收到对象池。

### 掉落判定

- 普通掉落：获得 1 倍金币。
- 中央奖励槽：获得 3-10 倍金币或收藏碎片。
- 两侧掉落：少量金币，主要用于维持反馈。
- 特殊币：由任务、广告奖励或升级解锁，不由付费随机抽取产生。

### 成长系统

- 投币速度：降低投币冷却。
- 推板力度：略微增加推币效率。
- 奖励槽倍率：提高中央槽收益。
- 离线收益：离线期间按机器等级给少量金币。
- 收藏册：收集固定碎片解锁主题或一次性奖励。

### 任务系统

- 每日任务：投币 N 次、掉落 N 枚、命中中央槽 N 次、看 1 次激励视频。
- 成就任务：累计掉落、累计升级、解锁主题。
- 赛季任务：通行证经验来源。

## 6. 经济模型

### 资源

- Coin：投币消耗资源，可自然恢复。
- Gold：游戏内升级货币，由掉落产生。
- Gem：高级货币，可通过内购、活动和少量任务获得。
- Ticket：广告票或活动票，用于固定兑换，不做随机抽奖。

### 基础参数建议

| 参数 | MVP 建议值 |
| --- | --- |
| 免费 Coin 上限 | 50 |
| Coin 恢复速度 | 1 枚 / 60 秒 |
| 单次投币消耗 | 1 Coin |
| 普通掉落收益 | 1-3 Gold |
| 中央槽收益 | 10-30 Gold |
| 激励视频奖励 | 30 Coin 或 5 分钟双倍 Gold |
| 每日激励视频次数 | 5-10 次 |
| 插屏间隔 | 至少 90-180 秒 |

### 付费点

- 6 元新手包：Gem + 限定硬币皮肤 + 去除部分插屏。
- 18 元月卡：每日 Gem + Coin 恢复加速。
- 30 元赛季通行证：固定奖励轨道。
- 皮肤包：机器主题、硬币主题、推板主题。

## 7. 技术方案

### 物理引擎约束

Web MVP 固定使用 `Phaser 3 + Matter.js`。不得切换到 Arcade Physics，因为推币机核心玩法依赖：

- 旋转硬币。
- 多硬币堆叠。
- 推板挤压。
- 摩擦力和空气阻力。
- 掉落槽精确判定。

Matter.js 初始配置建议：

```ts
const matterConfig = {
  gravity: { y: 1.1 },
  positionIterations: 8,
  velocityIterations: 6,
  constraintIterations: 2,
  enableSleeping: true
};
```

硬币刚体建议：

```ts
{
  restitution: 0.05,
  friction: 0.14,
  frictionAir: 0.025,
  density: 0.002,
  slop: 0.02
}
```

## 9. 美术和音频

MVP 可以先用简洁 2D 街机风：

- 台面：深色金属边框 + 透明玻璃高光。
- 硬币：金色圆形 sprite，带轻微旋转。
- 推板：金属块，周期性前后移动。
- 掉落反馈：金币飞向 HUD、数字飘字、轻微屏幕震动。
- 音效：投币、碰撞、掉落、中央槽命中、升级。

### Lovart 资产切片流水线

- 主视觉：已生成 `附件/推币机游戏/coin-pusher-hero.png`，用于方案、宣发和风格锚点。
- 硬币：Lovart 输出 512x512 金币、银币、特殊币，再压缩成 64x64 或 128x128 sprite。
- 硬币序列帧：制作 8-12 帧轻微翻转高光，用于投币和掉落反馈。
- 竖屏 UI 背景：生成 9:16 街机框体边缘背景，Phaser 中推币台居中，两侧用纹理填充。
- 主题皮肤：推币机外壳、推板、奖励槽、玻璃高光分层输出，方便后续换皮。
- TexturePacker：正式项目用 TexturePacker 或同类工具打包 atlas，减少 draw call。

## 13. 风险和对策

| 风险 | 对策 |
| --- | --- |
| 被认定为博彩 | 不做提现、实物兑换、付费随机奖励和概率宣传 |
| 物理手感不好 | 先做 Web 原型，快速调摩擦、质量、推板速度 |
| 广告打扰用户 | 激励视频为主，插屏加冷却和场景限制 |
| 数值通胀 | 升级成本指数增长，离线收益设上限 |
| 小游戏审核不通过 | 早期就使用合规命名、任务前置和固定奖励机制 |
| 被误判为博彩变种 | 首屏突出主线任务、图鉴收集和机器升级，弱化单纯投币表达 |
| Matter 物理穿透 | 推板碰撞体加厚、固定时间步、限制硬币速度和生成频率 |
| 移动端性能不足 | 硬币对象池、限制刚体数量、启用 sleeping、降低低端机硬币上限 |

## 14. 推荐下一步

先实现 Web 单机 MVP。项目结构建议使用 Vite + TypeScript + Phaser + Matter.js，3-5 天可以做出可试玩版本；确认手感和留存后，再决定是否迁移微信小游戏并接入广告/支付。
